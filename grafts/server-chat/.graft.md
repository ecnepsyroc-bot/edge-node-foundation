# Graft: Server ↔ Chat

## Purpose
Connects ServerManager (HTTP/WebSocket) to ChatManager (persistence).

## Bridges
- `ServerManager` (rami/server/manager.js)
- `ChatManager` (rami/chat/manager.js)

## Responsibilities
- Map HTTP endpoints to ChatManager methods
- Handle WebSocket message events
- Normalize message field names (legacy support)
- Broadcast database changes to all clients
- Provide authentication endpoint
- Handle CRUD operations for messages, subcategories, individual chats

## HTTP Endpoints

### Authentication
- `POST /api/login` - Register/login user

### Messages
- `GET /api/messages?job=X&subcategory=Y&individualChat=Z` - Get filtered messages
- `DELETE /api/messages?message_id=X` - Delete single message by ID
- `DELETE /api/messages?username=X&job=Y&subcategory=Z` - Delete user messages (excludes problems)

### Problems
- `POST /api/problems/:id/solution` - Mark problem as solved
- `DELETE /api/problems/:id` - Delete problem by ID

## WebSocket Events

### Incoming (from client)
- `authenticate` - Login with username
- `message` - Send new message

### Outgoing (to clients)
- `auth_success` - Login successful
- `message` - New message broadcast
- `message_deleted` - Message deleted
- `problem_created` - Problem created
- `problem_solved` - Problem solved
- `problem_deleted` - Problem deleted

## Data Normalization
`normalizeMessage()` ensures consistent field names:
- Accepts: `user` or `username`
- Accepts: `job` or `job_id`
- Accepts: `individualChat` or `individual_chat_id`
- Stores aliases for backward compatibility

## Does NOT Handle
- Job validation (delegated to chat-jobs graft)
- NLP processing (delegated to ui-nlp graft)
- UI rendering (in UI branch)

## Integration Pattern
```javascript
const serverChatGraft = new ServerChatGraft(serverManager, chatManager);
if (serverChatGraft.handleHTTPRequest(req, res)) {
  return; // Request handled
}
```
